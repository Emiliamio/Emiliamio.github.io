## 项目背景

大三暑假找实习前，我决定做一个能拿得出手的全栈项目。结合之前了解到的企业安全审计需求，选择做一个**日志审计查询系统**。

这个项目的目标很明确：不是做一个课程作业，而是模拟一个准生产级系统——有完整的前后端分离、有数据库索引优化、有异步处理、有操作审计自闭环。

## 技术选型

| 层级 | 技术 |
|---|---|
| 后端框架 | Spring Boot 3.2 + JDK 17 |
| 数据库 | MySQL 8 + MyBatis + HikariCP |
| 缓存 | Redis + Lettuce |
| 异步处理 | 自定义线程池 + @Async |
| 文件导出 | Apache POI (Excel) |
| 前端 | 原生 HTML + CSS + JavaScript（控制台风） |

选 Spring Boot 而不是其他框架的原因很简单：实习岗位的主流技术栈就是它。用 MyBatis 而不是 JPA 是因为日志查询需要多条件动态筛选，MyBatis 的动态 SQL 比 JPA 的 Criteria API 直观得多。

## 数据库设计——索引是核心

三张表：`user`（用户）、`log_entry`（日志记录）、`audit_log`（操作审计）。

`log_entry` 是最核心的表，数据量可能达到百万甚至千万级。索引设计花了最多心思：

- `idx_timestamp`：按时间范围查日志是最高频场景
- `idx_ip_address`：查"某个 IP 干了什么"是审计刚需
- `idx_timestamp_ip (timestamp, ip_address)`：**联合索引**覆盖了最高频的查询——"某段时间内某个 IP 的所有操作"。先按时间缩小范围，再按 IP 过滤，走 B+树的剪枝。

面试官问到索引时，我能说出每个索引对应哪个业务场景、联合索引为什么左列选时间而不是 IP——因为这些都是在写代码之前就想好的。

## 异步批量导入——线程池调优

日志审计系统的数据不是一条条手动加的，而是从 CSV 文件批量导入。同步导入会卡住整个请求，所以用 `@Async` 异步处理。

配置了自定义线程池：核心线程 4、最大线程 8、队列容量 100。为什么是这几个数？

我的电脑是 4 核 CPU，核心线程 = CPU 核心数，充分利用计算资源。日志导入是 IO 密集型——大部分时间花在磁盘读写和网络 IO 上，CPU 并不忙，所以最大线程翻倍到 8。队列 100 是一个合理缓冲，太大内存浪费，太小频繁触发拒绝策略。

拒绝策略选了 CallerRunsPolicy：线程池满了让提交者自己执行，实现自然反压。

## 操作审计——系统的自我闭环

这是我设计中最得意的一个点：**审计系统自己的操作也要被审计**。

每当你查日志、导出日志、导入日志，系统会自动在 `audit_log` 表里记一条——操作者是谁、查了什么条件、什么时间。如果有一天数据泄露，我们可以追溯是谁在什么时候查了哪批数据。

面试时讲这个点，可以自然引出对架构设计的思考——"系统不止审别人，也要审自己"。

## 前端：原生三件套也能好看

不想被前端框架绑架，用了原生 HTML + CSS + JavaScript 写了三个页面：日志列表、详情、统计面板。深色控制台风格，低蓝光配色，适合长时间盯着看。

前端代码放在 `resources/static` 下，Spring Boot 自动托管，浏览器打开就能访问，不需要额外部署。

## 收获与思考

这个项目让我完整走通了从需求分析、技术选型、数据库设计、后端开发、前端实现到部署上线的全流程。

最大的体会是：**写在简历上的项目，不在于功能多花哨，而在于面试官深问每一个点时，你都能答出"为什么这么设计"。**

索引为什么是联合索引而不是单列？线程池为什么是 4/8/100？拒绝策略为什么选 CallerRunsPolicy？操作审计表为什么要单独建而不是记日志表里？——每个选择背后都有原因，这些原因就是你的技术深度。
